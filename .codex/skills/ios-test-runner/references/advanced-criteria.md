# Advanced Criteria

仅在测试意图明确要求下列非终态证据时读取。先完成 `SKILL.md` 的业务终态判定；这些机制只补充证据，不能把过程信号提升为业务成功。

## 日志计数

用于验证“动作仅触发 N 次”等 UI 不可见事实。

1. 动作前调用 `app_logs_mark`，保存完整 `cursor`，并确认目标 source 的 `capture.state == "enabled"`。
2. 执行动作后调用 `app_logs_read(after:<完整 cursor>, sources:[...])`。
3. 若 `hasMore == true`，持续把完整 `nextCursor` 传给下一页；直到 `hasMore == false` 才能计数。
4. `gap` 非空表示证据不完整，应重新 mark 并缩短动作到读取的间隔。
5. 只有契约明确的 `source/category/message` 才能计数。`explore` 命令完成只证明命令链成功，不能代替业务结果；`bridge` 无条目不能证明代码未执行。

`notCaptured` 表示未启用，`unavailable` 表示请求过但运行环境不可用。两种空结果都不能证明“没有日志发生”。完整语义以 `ios-logs` 为唯一来源。

## 短暂中间态

spinner、按钮禁用等中间态只用于验证状态机本身，不得作为业务成功判据。动作后可立即用 `ui_wait(mode:"targetExists")` 或 `wait_and_inspect` 等待“出现”，确认出现后再等待“消失”；如果目标从未被观察为存在，不得用 `targetGone`，否则首轮会立即满足。

即使使用组合工具，也只能观察其固定采样点，不能保证捕获在一次工具往返内已经出现又消失的状态。测试目标要求这种短窗口且当前工具无法可靠观察时，结果标 `skipped` 并写明观测能力限制。

## Secure 字段保留

secure 字段不暴露原文。验证失败后是否保留时，优先使用 App 已公开的长度/掩码状态或业务日志契约；没有直接信号时才使用二次提交：

1. 第一次提交制造已知失败并等待明确终态。
2. 不重新输入 secure 字段，再次提交。
3. 同时等待“空字段即时校验失败”和“请求确已再次发起/异步终态”。前者命中说明字段已清空，后者命中说明字段仍可用于提交。

不得用“某提示没有出现直到超时”单独证明字段被保留；缺席可能来自未建模状态。也不得复用第一次失败后残留的同文案作为第二次请求证据。
