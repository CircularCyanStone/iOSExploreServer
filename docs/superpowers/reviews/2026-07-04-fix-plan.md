## 一、三方评估综合结论

### 1. 报告整体可信度：高，但有一处误报 + 一处严重低估

三方（报告 / codex / 主会话）交叉验证后，报告里 4 个严重问题（§3.1.1–3.1.4）与抽查的中问题（M8/M10）**事实部分全部成立**。但有 4 处必须修正：

### 2. 报告的 4 处缺陷

#### 缺陷 A（最危险）— M1 把「重复抽象」误报成「死代码」

- 报告原话：「`UIKitLocator` 生产代码 0 真实使用 → typealias 删 50 行」。
- rg 复核（排除定义文件）：**`UIKitLocator` 有 91 处引用**，`UIKitViewLookupTarget` 有 40 处。`UIKitActionPlan.tap/controlEvent` case、`UIKitLocatorResolver.locate/contains`、`UIKitActionExecutor.executeTap/executeControlEvent/locatorSummary`、`UIWaitExecutor` 全部直接持有 `UIKitLocator`。
- 真相：两类型 case 完全相同（`.accessibilityIdentifier(String)` + `.path([Int])`）、`logSummary` 逐字重复、`UIKitLocator.parse` 只是委托 `UIKitViewLookupTarget.parse` 再恒等映射。这是**重复抽象 + 桥接层**，不是死代码。
- **若按报告原方案直接删 `UIKitLocator`，会炸掉执行链路 6+ 文件的编译。** 盲目信任报告或单一工具（codegraph 的 caller 索引对「类型作为参数」追踪不全，也曾给出 "no callers" 的误导结果）都会踩坑。

#### 缺陷 B（真正最高优先级）— §3.1.4 被严重低估为「文案过期」

- 报告把 §3.1.4 描述为「文案过期 + code 用 `.invalidData` 不准」。
- 实际是**多方契约空洞**：`unsupported_target` 这个错误码已经被当成既定契约写进——
  - **6 处对外文档**：`docs/architecture/index.md:124`、`docs/tools/network-tools.md:119`、`docs/uikit/uikit-file-reference.md:119,131`、`docs/uikit/reading-guide.md:106`、`docs/superpowers/agent-mcp-exploration/README.md:179`、`agent-usage-protocol.md:153`、`curl-json-loop-protocol.md:207`
  - **3 处测试名**：`UIKitActionExecutorTests.swift:89,109`（`@Test("... 返回 unsupported_target")`）、`UIKitCommandErrorTests.swift:66`（函数名 `unsupportedTargetMapsToInvalidData`——名字本身就在记录「实现把 unsupported_target 映射成了 invalidData」这个偏差）
  - **5 处代码注释**：`UIKitActionExecutor.swift:18,115`、`UIKitDefaultActivationResolver.swift:32` 等
  - **Mac MCP 规划**：`docs/superpowers/specs/2026-07-03-mac-mcp-server-scope.md:26` 已把 `unsupported_target` 列为 iPhone envelope → MCP 的映射输入 code
- 但 `Sources/iOSExploreServer/Models.swift:147-225` 的 `ExploreError` enum **26 个 case 里根本没有 `unsupportedTarget`**，源码返回 `.invalidData`。
- 这是「设计意图里有、实现时漏补 enum case」，不是文案问题。影响：未来 Mac MCP 对接时，agent 拿不到 `unsupported_target` 按码分支决策，只能看到 `invalid_data`；测试还把这个偏差锁死了。

#### 缺陷 C — §4.2 方案 C 否决理由不严谨

- 报告写「XCTest `_XCEventGenerator` 真机不可用」。不准确：XCUITest 能由 Mac 侧 runner 驱动真机。
- 准确理由（codex）：`_XCEventGenerator` 是测试框架私有实现，不适合作为**App 进程内嵌 debug server** 的运行时依赖——会把命令能力绑死在测试 runner / XCTest 私有框架 / 开发者工具环境上，和「Mac curl → App 内 HTTP server → 真机 App 自执行」的闭环不一致。

#### 缺陷 D — 行号 / 章节小错

- 报告 `2026-07-04-refactor-review.md:75` 把 alert Release bug 标成「§3.1.3 的错误码 bug」，实际是 §3.1.1。
- §3.1.3 引用 `UIKitFingerprintCollector.swift:356-453`，当前 checkout 实际是约 `390-453`。
- M10 只点了 `UIScrollStepResult`，漏了 `UIScrollExecutor.swift:8-11` 同类过期注释（也说 scroll 原语供 scrollToElement 复用）。

## 二、分波次修复方案

### 第一波：契约修正 + 死代码清理（低风险，先做）

#### P1 — 补 `unsupported_target` 错误码契约（最高优先）

**为什么最先做**：6 文档 + 3 测试 + Mac MCP 规划已承诺这个 code，源码缺失是契约空洞；且第三波 `ui.tap realTouch` spike 的失败分流也要用它。先把契约补上，后续都顺。

改法：
1. `Sources/iOSExploreServer/Models.swift` `ExploreError` 新增 `case unsupportedTarget = "unsupported_target"`，注释说明「目标无默认激活路由（如 slider/segmented/普通 view 请求 tap），与参数校验失败的 `invalid_data` 区分」。
2. `Sources/iOSExploreUIKit/UIKitCommandError.swift:131-135` `unsupportedTarget` 工厂：code 从 `.invalidData` 改 `.unsupportedTarget`；message 改「目标没有默认激活路由（仅 UIButton / UISwitch / 文本输入支持默认 tap）」；注释删掉「第一版」「UIControl」措辞。
3. 测试同步：
   - `UIKitCommandErrorTests.swift:66-69`：函数 `unsupportedTargetMapsToInvalidData` 改名 `unsupportedTargetMapsToUnsupportedTarget`，断言改 `.unsupportedTarget`。
   - `UIKitActionExecutorTests.swift:89-104,109-124`：断言从 `.invalidData` 改 `.unsupportedTarget`（测试名本就写 unsupported_target，改后名副其实）。

验证：`swift test`（SPM）+ framework `xcodebuild ... test`。文档暂不用改（文档已是 `unsupported_target`，改完代码与文档对齐）。

**关键提醒**：报告原方案「删 UIKitLocator」会炸编译，不能照做。正确方向是「合并重复」。

改法（按风险递增，逐步验证）：
1. 先评估 `UIKitLocator` 能否 `typealias` 为 `UIKitViewLookupTarget`（case 完全相同）。typealias 是透明的，91 处引用自动有效。

### 第三波：基调迁移（高价值高风险，spike 驱动）

#### P7 — `ui.tap` realTouch spike（依赖 P1 完成）

**为什么放最后**：风险最高，依赖 P1 的 `unsupported_target` 错误码做失败分流。

落地节奏：
1. **先 spike**（参照 `Tests/.../UIAlertActionHandlerSpikeTests.swift` 模式，Debug-only 真机测试）：iOS 17/18/26 真机验证 codex 给的 4 件事——
   - ① 能否触发 `UITapGestureRecognizer`
   - ② 能否触发被 hit-test 命中的普通 view
   - ③ 透明遮挡时是否与真人一致命中遮挡层（验证不掩盖布局 bug）
   - ④ UIButton/UISwitch 旧默认模式是否仍兼容
2. **spike 通过** → 新增 `Sources/iOSExploreUIKit/Support/Runtime/UITouch+Synthetic.swift`（`#if DEBUG #if canImport(UIKit)`，参照 `UIAlertController+TriggerAction.swift` 既有隔离模式）+ `Support/Action/UIRealTouchExecutor.swift`；命令层加 `dispatchMode: "default" | "realTouch"`，`realTouch` 失败抛 P1 的 `unsupported_target`。`default` 保持现状，不改 `help` 主推荐路径，直到真机矩阵全绿。
3. **spike 失败 fallback（报告没给，本方案补）**：若 UITouch ivar 漂移导致不稳定，降级为「按 iOS 版本探测 ivar，支持版本走 realTouch，不支持版本回退 default + 返回 `unsupported_target`」，不强制全版本可用。
4. **历史背景**（codex 补充）：`docs/superpowers/specs/iOSExploreServer-ui-tap-final-refactor-plan.md:252-264,1038-1046` 旧设计文档写过 UITouch 合成不可靠；基调变了可从「不做」升为「允许 spike」，但不能跳过 spike。

#### P8 — `recursiveDescription` 增强（与 P7 配套，低优先）

让 `ui.topViewHierarchy` / `ui.viewTargets` 看到私有 helper view（`_UIButtonBarButton` 等）。与 realTouch 配套：点私有 helper view 需先能定位到它，二者有依赖，应配套推进。

### 持续约束

- 第三波私有 API 代码必须 `#if DEBUG`（+ `#if canImport(UIKit)`）隔离，参照 `Support/Runtime/` 既有模式。
- 私有 selector / ivar 名漂移是正常维护成本（`_dismissWithAction:` 已接受这类成本），按 iOS 版本适配，不作为拒绝理由——但合成 UITouch 的维护量级（十几个 ivar × N 版本）远大于单个 selector，spike 阶段就要评估这个长期成本。
- 每波完成三层验证：`swift test`（SPM，当前 210 个）+ framework `xcodebuild ... test`（当前 310 个）+ `Examples/SPMExample` 真实闭环（启动参数 `--ios-explore-autostart` 自动起 server）。
