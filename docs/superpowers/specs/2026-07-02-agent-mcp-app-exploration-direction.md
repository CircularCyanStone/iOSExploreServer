# Agent MCP 应用探索服务方向稿

> 日期：2026-07-02
>
> 本文记录当前项目基调、与得物 AI UITester 文章的关系、短期不做什么、后续设计应优先解决什么。它不是实施计划，也不是承诺要做一个完整测试平台。

## 1. 当前项目的真实目标

当前目标不是建设一个完整的“测试平台”。

更准确的目标是：

```text
做一个运行在 App 内的本地服务，
让外部 Agent 可以持续获取 App 状态、执行安全操作、拿到执行后的反馈，
从而一步一步探索和测试应用。
```

最终希望达到的使用效果是：

```text
开发者用自然语言描述一段测试案例
→ Agent 理解要验证的目标
→ Agent 通过 MCP 服务观察 App、操作 App、等待反馈
→ Agent 根据页面结果判断测试是否通过
```

也就是说，本项目不是只做一个“远程点点按钮”的工具。它要服务的是一个能按自然语言目标完成 App 操作和验证的 Agent。

但当前不做得物文章里那种更上层的平台化能力，例如用例平台导入、批量用例生成、执行报告系统、版本归档和大规模自愈工作台。这些能力以后可以接在外层，和原生端库没有直接耦合。

这里的 MCP 服务，可以理解为“Agent 和 App 之间的工具接口”。Agent 不直接碰手机内部对象，而是通过一组明确工具来：

- 看当前页面有什么；
- 找到可操作目标；
- 执行点击、输入、返回、滚动等动作；
- 等待动作后的页面结果；
- 再次观察页面，决定下一步。

因此，本项目近期最重要的不是平台化，也不是生成大量测试用例，而是先把这条闭环做好：

```text
观察页面
→ 判断下一步
→ 执行动作
→ 等待反馈
→ 再观察页面
→ 继续判断
```

## 2. 对未来测试平台保留余地，但当前不实现

得物文章里提到的用例平台、知识库、自愈、报告、版本归档都很有价值，但这些是更上层的系统。

当前只需要在架构上不要堵死未来：

- 后续可以接入用例描述，把自然语言用例转成步骤；
- 后续可以维护业务知识，例如“登录成功看哪个标记、失败看哪个提示”；
- 后续可以做失败诊断和自动修复；
- 后续可以生成报告和记录历史。

但当前阶段不把这些做成平台，也不把它们塞进 iPhone 端库里。

## 3. 从得物文章吸收什么

得物文章最值得吸收的，不是“所有操作都靠截图和视觉模型”，而是以下经验。

### 3.1 每次只走一步

UI 自动化最容易错在“预先想好很多步，然后一路执行”。真实页面每一步之后都可能变化：弹窗、加载、跳转、错误提示、登录失效、网络失败都可能出现。

所以 Agent 应该每次只做一个动作，然后重新观察页面。

### 3.2 点击成功不等于业务成功

`ui.tap` 成功只表示“这个点击动作已经发出去了”。

它不能证明：

- 页面已经跳转；
- 网络请求已经完成；
- 列表已经刷新；
- 登录已经成功；
- 业务真的完成。

所以点击之后必须有“等待反馈”的能力。

### 3.3 等待的是用户能看到的结果

不要默认用固定 sleep。

更好的方式是等页面上明确的结果，例如：

- 首页根视图出现；
- 错误提示出现；
- 重试按钮出现；
- 空状态出现；
- 登录失效弹窗出现。

这样 Agent 才知道下一步该继续、重试、处理弹窗，还是停下来。

### 3.4 失败要先分类

不能把所有失败都叫“点击失败”。

至少要区分：

- 设备或连接问题；
- 请求超时；
- 目标找不到；
- 目标过期，页面已经变了；
- 找到多个候选，不能确定点哪个；
- 控件不可用；
- 页面进入了业务失败状态；
- 弹窗遮挡。

分类清楚后，Agent 才能做正确选择，而不是盲目重试。

### 3.5 宁可不点，也不要点错

这是底层工具必须坚持的原则。

如果目标不确定、页面过期、控件不可用，工具应该明确失败，让 Agent 重新观察，而不是猜一个位置去点。

### 3.6 知识库要小而准

得物文章强调“错误知识比无知识更有害”，这个判断非常重要。

未来如果给 Agent 准备业务知识，应该是短小、明确、能验证的文档，例如：

```text
登录模块：
入口：profile.tab
提交按钮：login.submit
成功状态：home.root
失败状态：login.error
网络重试：network.retry
危险动作：发送验证码需要人工确认
```

不要一开始写一本大而全、很快过期的说明书。

### 3.7 自愈要有证据

未来如果做自动修复步骤，必须依赖失败截图、页面结构、错误类型、业务知识和置信度。

低置信度时应该让人确认，而不是让 Agent 自动乱改步骤。

## 4. 不照搬得物文章的地方

得物文章做的是跨 iOS、Android、HarmonyOS 的统一测试系统，所以它大量依赖截图和视觉模型。这对它是合理选择。

但当前项目只面向 iOS App 内部，并且已经能拿到 UIKit 的真实结构信息。

因此默认路线不应该是：

```text
每一步截图
→ 让视觉模型猜页面
→ 猜一个坐标
→ 点击坐标
```

当前项目默认路线应该是：

```text
读取 UIKit 结构化页面信息
→ 用稳定标识或快照路径定位目标
→ 通过安全动作执行
→ 等待结构化页面结果
→ 必要时才截图作为证据
```

截图可以保留，但它主要用于：

- 人工排查；
- 失败现场留档；
- 结构化信息不足时辅助判断；
- 未来给 Mac 侧视觉模型使用。

视觉模型不应该进入 iPhone 端库，也不应该绕过已有的定位、安全检查和过期检查。

## 5. 当前库可以调整，必要时可以推翻

当前库是探索过程中长出来的，不是从一开始就完整规划好的。因此后续评估时不需要为了兼容旧想法而保留不合适的设计。

可以调整或推翻的包括：

- 命令命名是否清楚；
- `ui.wait` 是否应该继续作为单条件等待；
- 是否需要新增一个“多结果等待并返回最终页面”的命令；
- 返回结构是否适合 Agent 连续使用；
- 文档是否把工具职责说清楚；
- 当前工具是否过早暴露了不该暴露的细节；
- 某些能力是否应该从 iPhone 端移动到 Mac 侧 Agent。

但推翻时要守住几条底线：

- core 仍不依赖 UIKit；
- UIKit 操作仍在主线程安全执行；
- 不确定时不乱点；
- 错误要可分类；
- 日志要能解释发生了什么；
- 文档要同步记录设计原因。

## 6. 当前更合理的分层

为了避免一开始就做成大平台，先按三层理解，但当前只重点做前两层。

### 6.1 手机里的工具层

也就是现在的 `iOSExploreServer` 和 `iOSExploreUIKit`。

它负责给 Agent 提供可靠事实：

- 当前页面结构；
- 可操作目标；
- 目标是否可点；
- 点击是否已经发出；
- 页面是否达到某个可见状态；
- 最终页面快照和可继续操作的目标。

它不负责：

- 理解业务目标；
- 生成测试用例；
- 自动修复步骤；
- 长篇业务推理；
- 跨平台统一。

### 6.2 Mac 侧 Agent 编排层

这一层还没有完整实现，但它是本项目真正要服务的对象。

它负责把手机工具串起来：

```text
看页面
→ 决定点哪里
→ 调用工具执行
→ 等待反馈
→ 判断是否成功、失败、需要重试或需要人工确认
```

这里可以使用业务知识、截图、模型推理，但它不能绕过手机工具层的安全规则。

### 6.3 未来测试平台层

这一层暂时只保留余地，不作为当前目标。

未来它可以负责：

- 从用例平台导入用例；
- 把描述性用例转成可执行步骤；
- 管理业务知识库；
- 失败后诊断和自愈；
- 生成测试报告；
- 归档版本历史。

## 7. 对当前 `ui.wait` 的重新定位

当前代码已经有 `ui.wait`，它可以等待单个条件，例如目标出现、目标消失、文本出现、页面变化或页面稳定。

这说明项目已经开始解决“点击后等待”的问题。

但得物文章提醒我们：真实流程经常不是一个结果，而是多个可能结果并存。

例如登录后可能出现：

- 登录成功，进入首页；
- 密码错误；
- 网络错误；
- 需要验证码；
- 被弹窗遮挡。

如果一次只能等一个条件，Agent 需要多次调用、自己拼装结果，容易复杂，也容易漏掉终态。

因此后续应重点评估是否需要一个更适合 Agent 的等待能力：

```text
等待多个可能结果
→ 命中哪个就返回哪个
→ 超时也返回最终页面状态
→ Agent 根据结果决定下一步
```

这个能力不一定必须叫 `ui.waitFor`，命名可以重新讨论；关键是语义要清楚。

## 8. 近期方向建议

近期不要急着做平台，也不要急着引入视觉模型。

更合理的顺序是：

1. 先把本项目定位写清楚：它是 Agent 使用 App 的工具服务，不是完整测试平台。
2. 重新审视现有命令，尤其是观察、动作、等待、截图这些能力是否能组成稳定闭环。
3. 明确点击、等待、业务成功三者的边界。
4. 设计一个 Agent 使用协议：每一步如何观察、如何选择动作、如何等待反馈、失败如何分类。
5. 再决定是否新增或改造“多结果等待并返回最终页面”的能力。
6. 最后才讨论知识库、视觉模型、自愈和测试平台化。

## 9. 本次讨论确定的关键点

- 当前目标不是 Agent Tester 平台，而是 Agent MCP 应用探索服务。
- 最终效果仍然是：开发者用自然语言描述测试案例，Agent 借助 MCP 服务完成 App 操作并验证结果。
- 平台化能力可以保留设计余地，但当前不实现。
- 用例平台、报告、版本归档、大规模自愈工作台属于更上层能力，后续可扩展，但当前不放进原生端库目标。
- 得物经验要认真吸收，但不能照搬它的跨端视觉路线。
- 当前代码没有历史兼容负担，不合理的方向可以调整甚至推翻。
- 关键讨论节点和重要判断必须写入文件，不能只留在对话里。
- 文档要尽量用普通开发者能理解的话写，必要术语必须解释。
